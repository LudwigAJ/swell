# Autonomous Coding Engine: Technical Architecture and Roadmap Spec

**The optimal architecture for a long-running autonomous coding engine is a headless, event-sourced daemon with protocol-separated frontends, hierarchical multi-agent orchestration, and a phased build sequence that starts CLI-first and defers custom editor investment until product-market fit is proven.** This conclusion is drawn from analyzing the architectures of 16 current agentic coding products, 8 multi-agent orchestration frameworks, and the build sequences of every major player in the space. The most critical architectural insight is that context engineering — how the system retrieves, ranks, and assembles relevant code context — is the primary technical differentiator, not the UI surface or the agent framework. The architecture must optimize for replaceability at the model layer, durability at the state layer, and composability at the protocol layer.

The competitive landscape is instructive. Cursor reached **$2B+ ARR** by forking VS Code and building a custom context engine. Claude Code reached **$2.5B ARR** in roughly one year with a CLI-first, headless design. Devin demonstrated that cloud-hosted full autonomy works for well-scoped tasks but struggled with broader adoption. Every successful product followed the same sequence: interactive assistance first, agentic capabilities second, full autonomy third. The architecture proposed here is designed to support all three modes from the foundation up, without prematurely committing to a UI surface or a specific orchestration framework.

## The reference architecture: five layers and three protocols

The system decomposes into five distinct layers, each with clear boundaries and communication protocols. This decomposition is informed by how Docker separates its daemon from its CLI, how LSP separates language intelligence from editors, and how Claude Code separates its agent core from its multiple frontends (CLI, VS Code, JetBrains, desktop, web, mobile — all connecting to the same engine).

**Layer 1 — Operator Surface** handles all human interaction. This is deliberately the thinnest layer: a CLI, an IDE extension, a web dashboard, or eventually a desktop application. Each surface is a protocol client that connects to the engine via the same API. Claude Code proves this pattern works — its CLI, VS Code extension, JetBrains plugin, desktop app, and web interface all share the same underlying engine. The operator surface never contains business logic; it renders state, captures input, and streams events.

**Layer 2 — API/Event Gateway** is the protocol boundary between surfaces and the engine. It exposes a JSON-RPC interface (matching LSP/ACP conventions) over Unix domain sockets for local communication and WebSocket for remote. All state changes flow as structured events. This layer handles authentication, rate limiting, and session management. It supports the emerging Agent Client Protocol (ACP), built by JetBrains and Zed, which standardizes agent-to-editor communication the same way LSP standardized language servers.

**Layer 3 — Orchestrator** is the brain. It receives goals (PRDs, feature briefs, prompts), decomposes them into tasks, spawns and supervises worker agents, manages the execution DAG, handles failures and retries, and maintains the planning state. Research from Google DeepMind (Kim et al., December 2025) is definitive here: **centralized coordination reduces error amplification from 17.2x to 4.4x** compared to flat/independent agent topologies. The orchestrator uses a Planner-Worker pattern where a frontier model plans and cheaper models execute — a pattern proven at Devin, Cursor, and Klarna to achieve **up to 90% cost reduction** versus using frontier models for everything.

**Layer 4 — Execution Runtime** is where agents actually run. Each agent operates in an isolated environment (Docker container for development, Firecracker microVM for production) with its own filesystem, terminal, and tool access. Agents communicate with the orchestrator through the event stream but cannot directly affect each other's state. This is the pattern used by OpenHands (Docker sandboxing), Devin (cloud VMs), and E2B (Firecracker microVMs booting in ~80ms).

**Layer 5 — Persistence and Knowledge** stores everything: agent state (event-sourced), code context (vector + graph indexes), session history, configuration, and checkpoints. This layer uses SQLite/libSQL for structured state, LanceDB for vector search, and the filesystem (via git) for code artifacts. All state is local-first with optional cloud sync.

Three protocols bind these layers together. **ACP** (Agent Client Protocol) connects operator surfaces to the engine, following the LSP model of capabilities negotiation over JSON-RPC. **MCP** (Model Context Protocol) connects the engine to external tools and data sources — now an industry standard under the Linux Foundation with **5,800+ servers and 97M+ monthly SDK downloads**. An internal **event protocol** connects the orchestrator to worker agents via structured, append-only event streams. These three protocols ensure that any layer can be replaced independently without architectural rework.

## Engine core: why the language choice matters less than the architecture

The question of Rust versus Go versus TypeScript for the engine core is the most frequently debated and least consequential architectural decision. The data tells a nuanced story.

**Rust** delivers the best raw performance: Zed's Rust-based editor achieves **0.12s cold startup** versus VS Code's 1.2s, uses under **100MB RAM** versus VS Code's 200-400MB, and renders at 120 FPS via GPU acceleration. Rust's ownership model prevents data races at compile time — critical for a system managing concurrent agent sessions. The ecosystem is maturing rapidly: Tree-sitter, gitoxide, Tokio (async runtime), and LanceDB are all Rust-native. However, Rust's steep learning curve and longer compile times slow development velocity. Hiring is scarce and expensive.

**Go** offers the best pragmatic tradeoff for infrastructure software. Docker, Kubernetes, containerd, Terraform, and Prometheus are all written in Go. Microsoft chose Go over Rust for the TypeScript compiler rewrite, achieving **10x faster compilation** — the team cited Go's simpler mental model and faster development velocity. Go's goroutines provide excellent concurrency for I/O-heavy workloads like LLM API calls and file operations. Hiring is plentiful, particularly in the infrastructure/cloud-native talent pool.

**TypeScript** is the language of the existing editor ecosystem. VS Code, Continue.dev, and Claude Code's agent harness are all TypeScript. The npm ecosystem provides the largest collection of developer tool libraries. However, TypeScript's single-threaded event loop and higher memory footprint make it less suitable for a long-running daemon managing concurrent agents.

The recommended approach is a **hybrid architecture** inspired by Lapce's proxy model and Zed's extension system. The core daemon — process supervision, IPC, file watching, sandboxing, and event streaming — benefits from a systems language (Rust or Go). The orchestration and agent logic layer, which changes more frequently and requires rapid iteration, can be TypeScript or Python. The connection between them uses gRPC over Unix domain sockets, achieving **sub-100µs latency** with strongly-typed interfaces. This is exactly how Docker works: the daemon is Go, communication with containerd is gRPC, and the CLI is a thin client.

The critical design principle is **not to couple the language choice to the architecture**. If the daemon protocol is well-defined, the engine can start as a TypeScript monolith for speed and have performance-critical paths rewritten in Rust or Go as profiling reveals bottlenecks. Premature optimization of the language choice is itself an overengineering risk.

## Daemon architecture: lessons from Docker, LSP, and OTP

The engine should run as a local daemon process, similar to `dockerd`, that starts on demand and remains running in the background. This pattern is well-established in developer tooling and provides several properties essential for an autonomous coding system.

**Process supervision** follows the Erlang/OTP model, adapted for the coding domain. A root supervisor manages three categories of child processes: permanent workers (file watcher, IPC server, LSP client manager), transient workers (agent sessions that restart on abnormal exit), and temporary workers (one-shot tasks like indexing jobs). The "let it crash" philosophy is appropriate — rather than defensive error handling in every agent, the supervisor detects failures via heartbeats and restarts agents with clean state from the last checkpoint. This maps to a clear state classification: static state (configuration, project structure) is recoverable from disk; recomputable state (indexes, embeddings) can be regenerated; irreplaceable state (conversation history, planned changes) must be persisted immediately to the event store.

**IPC choices** should follow established patterns. For daemon-to-frontend communication on the same machine, **gRPC over Unix domain sockets** provides the best balance of performance (15-50% higher throughput than TCP), type safety (Protobuf schemas), and schema evolution. For editor integration, **JSON-RPC over stdio** matches the LSP/ACP convention and ensures compatibility with VS Code, JetBrains, Zed, and Neovim. For LLM backend communication, standard HTTPS with streaming (SSE) is the only practical choice given how model APIs work. For real-time agent coordination within the engine, an internal event bus using either gRPC bidirectional streaming or in-process channels (depending on whether agents run in-process or as separate processes) handles the throughput requirements.

**Long-running session management** is the most novel aspect of this daemon compared to existing developer tool daemons. Unlike an LSP server that responds to requests, this daemon must maintain autonomous sessions that may run for hours. Each session requires its own checkpoint timeline, its own resource budget (token limits, API rate limits), its own sandbox, and its own git worktree. The daemon must support multiple concurrent sessions without resource contention. The OpenAI Symphony project (March 2026) provides an interesting reference: an Elixir/OTP-based orchestrator with GenServer processes per issue, lifecycle states (Unclaimed → Claimed → Running → RetryQueued → Released), and per-issue workspace directories with hooks.

## Operator surface strategy: build the engine first, frontends second

The most important finding from studying the competitive landscape is that **the agent is becoming a portable capability, decoupled from any specific editor**. Junie moved from JetBrains-only to standalone CLI. Claude Code works across seven surfaces. Cursor released ACP for cross-IDE support. Factory's Droids work from any interface. The architecture must treat the UI surface as a replaceable, thin client from day one.

**CLI-first is the correct starting point.** Claude Code's success validates this approach. A CLI tool built with a framework like React Ink (for terminal rendering) or a simple REPL validates the core agent loop, tool ecosystem, and execution model with minimal UI investment. The CLI naturally supports composability (`cat spec.md | engine plan`), CI/CD integration, scripting, and parallel instance execution. It also builds developer trust through transparency — every action is visible in the terminal.

**A VS Code extension is the correct second surface.** The extension API, while limited, supports webview panels, chat participants, terminal integration, and the Language Model Tools API. Extensions like Cline (5M+ installs) and Continue.dev prove that significant agentic capabilities fit within extension constraints. The critical limitation is that extensions **cannot modify the core editor chrome**, control the application layout, or implement custom inline completion flows — which is why Cursor and Windsurf forked VS Code. For an initial product, these limitations are acceptable. If they become blocking, a fork can be considered later.

**A desktop application is the correct third surface, if needed.** Tauri 2.0 is the recommended framework over Electron for any new project in 2026. The performance gap is dramatic: **97% smaller binaries** (3-10MB vs 80-150MB), **80% less memory** (30-50MB vs 200-300MB), and sub-0.5s startup versus 1-2s. Tauri's Rust backend provides native performance for heavy operations, while its webview frontend (using any web framework) provides familiar UI development. The main risk is cross-platform rendering inconsistency, since Tauri uses the system's native WebView rather than bundling Chromium. For code editing within a Tauri app, CodeMirror 6 is preferred over Monaco: it's **modular** (300KB core vs Monaco's 5-10MB), has excellent mobile support, and its functional/immutable state model works better across webview boundaries.

**The custom editor question** should be deferred indefinitely. The evidence is overwhelming. Zed, built by the creators of Atom and Tree-sitter with 15+ years of editor expertise, took **3+ years** to reach usability and still has a smaller user base than Cursor. Lapce remains pre-production after years of development. JetBrains Fleet was discontinued. Cursor forked VS Code in 3 months and reached $2B ARR. **The engineering cost of a custom editor is 3-5 engineer-years to reach VS Code parity** — time directly subtracted from AI and agent capabilities, which is where differentiation actually lives. Even Cursor, at $2B+ ARR, is only now beginning Cursor 3 ("Glass"), a new agent-management-focused interface built from scratch. The decision to build a custom surface should be driven by proven product-market fit, not architectural ambition.

## State, storage, and memory: the event-sourced foundation

State management is where autonomous coding systems diverge most sharply from traditional developer tools. A coding assistant that responds to requests needs only conversation history. An autonomous engine that runs for hours, spawns multiple agents, manages branches, and must survive crashes needs a fundamentally different approach.

**Event sourcing is the correct foundation pattern.** A February 2026 paper on Event Sourcing for Autonomous Agents (ESAA) formalizes what LangGraph, OpenHands, and Symphony have converged on independently: the source of truth is not the current state but an **immutable, append-only log of intentions, decisions, and effects**. Current state is a projection derived from replaying the event log. This provides crash recovery (replay from any point), audit trails (every decision is recorded), time-travel debugging (inspect any historical state), and concurrent agent serialization (append-only semantics naturally order concurrent activities). Combined with CQRS, the write path (appending events) and read path (materialized views for UI, analytics, debugging) can scale independently.

The storage stack should be layered. **SQLite in WAL mode** is the primary structured store, proven in VS Code (which stores extension state in `state.vscdb` files) and broadly in local-first applications. The recommended pragmas — `journal_mode=WAL`, `synchronous=NORMAL`, `cache_size=-64000` — provide concurrent reads with a single writer, which matches the agent event-append pattern. For projects requiring concurrent writes or cloud sync, **libSQL/Turso** adds embedded replicas and experimental `BEGIN CONCURRENT` support. For analytical queries over execution data (cost analysis, performance profiling), **DuckDB** operates as an embedded OLAP engine that can query event logs directly.

**Vector search for code-aware retrieval uses LanceDB** — an embedded, Rust-based, columnar vector database with zero-copy access and no server requirement. It's proven in Continue.dev for codebase indexing and handles larger-than-memory datasets efficiently. For code embeddings, **Voyage Code-3** achieves **97.3% MRR** on code search benchmarks, dramatically outperforming general-purpose models. As a local/open-source fallback, UniXcoder is the best BERT-family model for code retrieval. The critical implementation detail is chunking strategy: code must be chunked along semantic boundaries (functions, classes, methods) using Tree-sitter, not arbitrary token counts. Naive fixed-length chunking degrades embedding quality substantially.

**Context assembly is the highest-leverage engineering investment.** Cursor's founders have stated that their custom context retrieval model — trained specifically for ranking which code snippets are relevant — is core intellectual property and "entirely self-developed." The architecture should support a tiered context pipeline: active file (full content) → open/recent files → vector search results → graph-expanded context (callers, dependencies, type hierarchies) → conversation history. Each tier is scored and truncated to fit within the token budget. The key finding from production systems is that **long context did not kill RAG**: at a 2025 ICML benchmark (LaRA), RAG remained more cost-effective (~$0.00008/query vs $0.10-$10 for full-context), lower latency (~1s vs 30-60s), and better at handling dynamic codebases. The production default is hybrid: RAG for precision retrieval, long context for active session state.

**Git is the universal safety net and the primary code state manager.** Every successful tool uses git as the foundation for change management. Aider auto-commits every AI change with descriptive messages. Claude Code uses diff-based workflows with revertible changes. Devin and Factory deliver all results as pull requests. For parallel agent work, **git worktrees** (`git worktree add`) provide true isolation — each agent gets its own working directory on its own branch without filesystem conflicts. Programmatic git access should use `git2-rs` (libgit2 Rust bindings) for stability today, with a forward path to `gix` (gitoxide, pure Rust) as Cargo itself is migrating to it.

## Multi-agent orchestration: hierarchical, bounded, and checkpoint-recoverable

The research on multi-agent coordination reveals sharp constraints that should discipline the architecture. Google DeepMind's quantitative study (180 configurations, 5 architectures, 3 LLM families) found that **independent agents amplify errors 17.2x** while centralized coordination contains amplification to 4.4x. Coordination gains **plateau beyond 4 agents** — above this, coordination overhead consumes benefits. For sequential reasoning tasks, **every multi-agent variant degraded performance by 39-70%** compared to a single agent. These findings dictate a hierarchical topology with bounded parallelism, not an unbounded swarm.

The orchestrator implements a **Planner-Worker pattern with checkpointed state**. A frontier model (Claude Opus, GPT-5) receives the goal, analyzes the codebase, and produces a structured execution plan as a DAG of tasks with dependencies. Worker agents, running cheaper/faster models, execute individual tasks within sandboxed environments. The orchestrator monitors progress, handles failures (retry → model fallback → human escalation), and updates the plan as results arrive. This is the pattern used by Devin 2.0's Interactive Planning, Cursor's two-agent architecture, and OpenHands' AgentDelegateAction.

Task decomposition must respect the **35-minute degradation window** — agents show measurable performance degradation after approximately 35 minutes of equivalent human work. Tasks should be chunked to fit within this window, with verification gates every 3-5 sequential steps. The verification gate runs the test suite, static analysis (ESLint, Ruff, mypy, clippy as appropriate), and type checking after each chunk. If verification fails, the agent iterates; if it fails after 3 retries, the task is escalated.

**Failure handling uses a four-layer stack**, ordered by implementation priority. Layer 1: retry with exponential backoff and jitter (reduces retry storms by 60-80% per AWS research). Layer 2: model fallback chains (primary model → cheaper alternative → cross-provider fallback). Layer 3: error classification and routing — separate transient errors (retry), LLM-recoverable errors (reformulate prompt), and human-required errors (escalate). Layer 4: checkpoint recovery — restore from last successful checkpoint and re-execute only failed nodes. LangGraph's PostgresSaver provides a reference implementation for production checkpointing with thread-based state persistence.

**Agent spawning should be dynamic but depth-limited.** Research on AgentSpawn (2026) shows that dynamic spawning based on complexity metrics achieves **34% higher completion rates** than static agent topologies on SWE-bench, but spawn depth beyond 3 levels causes coordination overhead to exceed specialization benefits. Resource management for concurrent agents requires centralized rate limiting (not per-agent independent rate limiting, which causes thundering herd problems), priority-based API quota allocation (blocking architectural decisions get priority over background polling), and hard per-agent token budgets with circuit breakers. The MAST taxonomy of multi-agent failure modes identifies that **36.9% of failures are coordination breakdowns** — making explicit, typed state contracts at every agent boundary essential.

## Extensibility, sandboxing, and the security imperative

**MCP is the correct extensibility protocol** and should be adopted as a foundational architectural element. Since Anthropic donated it to the Linux Foundation in December 2025, MCP has achieved near-universal adoption: Cursor, Windsurf, Claude Code, Zed, Sourcegraph, Replit, Amazon Q, and JetBrains all support it. The protocol works on a client-server model over JSON-RPC 2.0, exposing tools (callable functions), resources (data sources), prompts (templates), and the newer Tasks primitive (async operations). For 2026, the roadmap priorities are transport scalability (Streamable HTTP), agent-to-agent communication, and enterprise readiness (audit trails, SSO). Current limitations include security concerns around prompt injection through malicious server responses and OAuth token exposure — the architecture should validate all MCP tool outputs before acting on them.

**Plugin isolation should use WebAssembly (Component Model).** Zed's production implementation provides the reference: extensions compile to `wasm32-wasip2`, run in the Wasmtime engine, define APIs via WebAssembly Interface Types (WIT) files for versioned contracts, and cannot crash the host process. Extism extends this pattern with host SDKs in 11 languages and manifest-controlled permissions (URL allowlists, filesystem mapping, memory limits, timeouts). This is materially more secure than VS Code's extension model, where extensions run in a Node.js process with filesystem and network access and have historically been vectors for supply chain attacks.

**Sandboxing for agent-generated code execution is non-negotiable.** The security threat landscape is severe and well-documented: Claude Code wiped a user's entire home directory via a trailing `~/` in `rm -rf`; Cursor deleted 70 files despite explicit instructions not to run anything; a nation-state group used Claude Code for cyber espionage with AI executing 80-90% of the attack chain; **48% of AI-generated code contains security vulnerabilities**. The sandbox hierarchy, from strongest to weakest isolation:

- **Firecracker microVMs** provide hardware-enforced isolation with ~125ms boot time, <5 MiB memory overhead, and up to 150 VMs/second/host. E2B uses Firecracker and reports sandbox startup in ~80ms. This is the gold standard for production.
- **gVisor** provides a user-space kernel that intercepts syscalls without touching the host kernel, with 10-30% overhead on I/O-heavy workloads. Appropriate for Kubernetes-native deployments.
- **Docker containers** share the host kernel and are considered insufficient for untrusted code by security consensus — "a permissive container running untrusted LLM-generated code is easily escaped." Docker Desktop 4.60+ runs containers inside dedicated microVMs, which addresses this, but plain Docker is appropriate only for development/testing.
- **WebAssembly/WASI** provides microsecond cold starts and capability-scoped isolation but cannot run shell commands, package managers, or most development tools. Appropriate for plugins, not for full agent sandboxing.

The recommended approach: use Docker containers for local development (acceptable risk for developer's own machine), Firecracker/E2B for production and cloud-hosted execution, and WASI for plugin sandboxing. Every agent sandbox should enforce filesystem isolation (mount only the project directory, never the home directory), network isolation (default deny, explicit allowlists), resource limits (CPU, memory, time), and secrets isolation (never inject credentials into agent context; use a vault with scoped, session-rotated tokens).

## Validation, observability, and cost control

Autonomous coding systems must validate their own output because **over 50% of AI-generated code contains logical or security flaws** and over 70% of developers routinely rewrite AI-generated code. Validation must be in the inner loop — after every code change iteration, not just at the end.

The validation pipeline runs in sequence after each agent edit: execute the test suite (existing tests + agent-generated tests) → run static analysis (ESLint, Ruff, mypy, clippy) → run the type checker → check for new warnings versus baseline → compute a confidence score. If any step fails, the agent iterates. The **Test-Driven Agent Development pattern** — where the agent writes tests from acceptance criteria first, then implements until tests pass — produces the most reliable results. Cursor's documentation explicitly recommends this: "Agents perform best when they have a clear target to iterate against." The confidence score aggregates test pass rate, linter warning delta, type error count, and build success into a metric that determines whether changes auto-merge or route to human review. Weekly recalibration of the confidence threshold based on observed error rates turns the approval queue into a self-improving system.

**Human-in-the-loop is implemented as confidence-based routing.** High-confidence outcomes (all tests pass, no new warnings, low-risk file changes) auto-merge. Low-confidence outcomes (test failures, security-sensitive files, infrastructure changes) route to human review. The architecture should define a clear permission boundary: reading files, running tests, running linters, creating files, editing project files, and committing to feature branches are autonomous; deleting files, installing dependencies, modifying CI/CD, accessing secrets, pushing to main, and database schema changes require approval. This boundary should be configurable per project via a rules file (analogous to CLAUDE.md or .cursorrules).

**Observability uses OpenTelemetry GenAI Semantic Conventions** (standardized attributes for model, tokens, provider, latency, cost) with Langfuse as the recommended open-source platform (MIT-licensed, 19K+ GitHub stars, OTel-native). Every agent action should emit a wide-format structured event capturing model, tokens, cost, tool invocation, confidence, and outcome. The critical metric is **cost per outcome**, not cost per request — an agent that takes 10 API calls to correctly resolve a bug is more valuable than one that takes 2 calls and fails.

**Cost control follows a hierarchy of interventions.** Prompt caching provides the highest ROI with the least effort: Anthropic's implementation saves **90% on cache reads**, and the pattern is straightforward — place stable content (system prompts, tool definitions, project rules) at the beginning of the prompt and append dynamic content at the end. Model routing provides the next tier of savings: a lightweight intent classifier routes simple tasks to GPT-4o-mini/Haiku ($0.15/M tokens) and complex tasks to Opus/GPT-5 ($1.25+/M tokens); if 60% of queries use the small model, average cost drops 50-70%. Semantic/plan caching — extracting reusable plan templates from completed executions — reduces cost by **50.31%** and latency by 27.28% in research benchmarks. Context compression using tools like LLMLingua achieves 20x compression with 1.5% performance loss. Combined, these strategies achieve **70-90% total cost reduction** from naive baselines. For a heavy user running 50 Claude Code sessions/day, the difference between optimized and unoptimized is roughly $1,000/month versus $5,600/month.

## Phased build strategy: from CLI agent to autonomous platform

### Phase 0 — Foundation (weeks 1-6)

The goal is to establish the technical substrate that everything builds on. Build the headless engine daemon with: an LLM abstraction layer supporting Claude, GPT, and Gemini via unified API; a tool execution framework (file read/write, shell execution, search, git operations); an event-sourced state store on SQLite; Docker-based sandbox for agent code execution; Tree-sitter integration for code parsing; basic codebase indexing (file tree, symbol extraction). Ship a **CLI as the primary interface** — a terminal REPL that accepts natural language, executes multi-step plans, and commits results to git. This validates the core agent loop with zero UI investment.

What to build permanently: the LLM abstraction layer, event protocol, sandbox infrastructure, and telemetry. What to build as replaceable: the initial codebase indexer (will need a custom ML model later), CLI rendering, and prompt templates.

### Phase 1 — Useful coding assistant (weeks 7-16)

The goal is a product that developers prefer over their current setup. Add: a VS Code extension with chat interface and inline diff preview; multi-file editing with accept/reject per change; codebase-aware context retrieval (vector search via LanceDB + grep/ripgrep); terminal integration for running commands; git integration with auto-commits; `.rules` file support for project-specific instructions; model routing (simple tasks to cheap models, complex to frontier). The key metric: can a developer use this for daily work and prefer it to VS Code + Copilot?

### Phase 2 — Agentic capabilities (weeks 17-28)

The goal is autonomous multi-step execution that justifies premium pricing. Add: Agent Mode (task decomposition, plan/act loop, self-correction); background agents running in git worktrees; session memory and project context persistence across sessions; MCP integration for external tools (Jira, Slack, databases); multi-agent orchestration with the Planner-Worker pattern; human-in-the-loop approval gates with confidence-based routing; cost tracking and budget controls. Decision point: add cloud-hosted agent execution (increases infrastructure burden but enables long-running autonomous work).

### Phase 3 — Autonomous platform (weeks 29-40+)

The goal is true long-running autonomous execution with minimal intervention. Add: event-triggered automations (CI events, Slack messages, calendar, timers); multi-agent specialization (code agent, test agent, review agent, documentation agent); custom context retrieval model trained on usage data; agent-to-agent coordination for cross-cutting tasks; enterprise features (SSO, audit logs, compliance, team management); optional desktop application (Tauri 2.0) for agent supervision and multi-session management. Decision point: evaluate whether to build a dedicated agent management console (the direction Cursor 3 is taking, where the IDE becomes secondary to the agent supervision interface).

### What differentiates early versus what can be deferred

Differentiate early on **context quality** (how well the system retrieves relevant code), **memory across sessions** (most tools still forget everything between sessions — this is a top developer frustration), **multi-file coherent editing** (the core skill separating good tools from great), and **cost transparency** (every competitor has stumbled on pricing). Defer: custom editor UI, enterprise compliance, deployment infrastructure, team collaboration, and custom model training. These are all important but none are blocking for initial adoption.

## Risks, tradeoffs, and open questions

**The custom IDE trap is the highest-impact risk.** Building a custom editor is a 3-5 engineer-year investment to reach VS Code feature parity, with no guarantee of adoption. Zed (built by Atom's creators, Rust, GPU-rendered) took 3+ years. Fleet (built by JetBrains) was discontinued. Lapce remains pre-production. Cursor reached $2B ARR by forking VS Code in 3 months. The evidence is unambiguous: **do not build a custom editor until product-market fit is proven at scale.** The correct strategy is to build the engine as a headless daemon, ship CLI + VS Code extension, and invest all engineering capacity in context quality, agent reliability, and orchestration.

**False autonomy risk requires honest calibration.** SWE-bench Verified scores (~80%) are massively inflated compared to real-world performance; SWE-bench Pro (private, truly unseen codebases) shows only **15-18% resolve rates**. Agents are reliable at well-defined bug fixes, scaffolding, test generation, documentation, and repetitive migrations. They are unreliable at ambiguous tasks, architectural decisions, security-critical code, and cross-cutting concerns. The architecture must implement an "autonomy slider" — letting users control how much independence the agent has — and never market as "fully autonomous." Devin's 67% PR merge rate on well-defined tasks represents a realistic ceiling for delegation.

**Model vendor lock-in is an existential risk for any product built on LLM APIs.** Anthropic (Claude Code), OpenAI (Codex), and Google (Antigravity) are all direct competitors. Building on their APIs means building on a competitor's platform. The architecture must abstract the model layer completely, support model switching at the task level, and invest in custom models for specific sub-tasks (context retrieval, diff prediction, code completion) that don't depend on any single provider. Open-source model quality is approaching closed-model performance on coding benchmarks (Qwen3-Coder, Devstral, GLM-5), making local model support increasingly viable.

**Infrastructure burden for local-first architecture** is real but manageable. Local execution means the engine competes for resources with the developer's other tools, cannot run when the machine is off, and cannot scale beyond local compute. The recommended mitigation is to start local-first (minimizes ops burden, builds enterprise trust for privacy/compliance) and add cloud execution as an opt-in capability in Phase 2. Never force cloud-only — the privacy argument is decisive for enterprise adoption.

**Overengineering is most likely in three areas**: the agent orchestration framework (start with a simple while-loop like Claude Code's, not a complex graph), the context retrieval pipeline (start with grep + vector search, add graph traversal later), and the storage architecture (start with SQLite, not a distributed database). **Shortcuts are most dangerous in three areas**: execution sandboxing (one escaped sandbox is catastrophic), diff application (incorrect code modifications destroy trust instantly), and git integration (lost code means lost users permanently).

### Open questions the industry has not resolved

The optimal level of human-in-the-loop engagement remains contested — Cursor favors continuous involvement, Devin favors delegation, Claude Code favors terminal autonomy. No clear winner has emerged. The pricing model is unsettled — per-seat (Cursor), per-compute-unit (Devin), per-token (Cline), and effort-based (Replit) have all caused user friction. Agent memory architecture lacks consensus — whether "files are all you need" (CLAUDE.md approach) versus structured memory systems (Windsurf's Memories, Augment's persistent context) is unresolved. Multi-agent coordination has no standard protocol for how agents hand off work, resolve conflicts, or merge outputs — MCP handles tool integration but not agent-to-agent coordination, and Google's A2A protocol is early-stage. External factors with the largest architectural impact are model capability trajectory (improving ~2x/year, making today's architectural accommodations potentially obsolete), cost trajectory (dropping rapidly, reducing the importance of token optimization), EU AI Act high-risk obligations (effective August 2026), and open-source model quality gains (making local-first increasingly viable).

## Conclusion: the architecture is the sequencing

The most important insight from this research is that the right architecture for an autonomous coding engine is not a static blueprint — it is a sequencing strategy. The products that won built thin, composable layers in the right order: core agent loop → context engine → interactive editing → agentic execution → multi-agent orchestration → custom surfaces. The products that struggled built ambitious systems in the wrong order, investing years in custom editors or complex orchestration before proving that their agent could reliably edit code.

The architecture proposed here — a headless, event-sourced daemon with protocol-separated frontends, hierarchical multi-agent orchestration, and hybrid retrieval — is designed for this sequencing. Every component is replaceable. The CLI can evolve into a VS Code extension, then a desktop app, then a custom surface, without architectural rework. The single-agent loop can evolve into Planner-Worker, then multi-agent orchestration, without changing the event store or the sandbox layer. The SQLite store can migrate to PostgreSQL, the Docker sandbox can upgrade to Firecracker, and the LLM provider can change quarterly — all without touching adjacent layers.

Three bets are embedded in this architecture that deserve explicit acknowledgment. First, that **context engineering will remain the primary differentiator** even as models improve — better models make better context more valuable, not less. Second, that **the agent management console will eventually displace the IDE as the primary interface** for software development, and that building the engine as a headless daemon positions for this transition. Third, that **local-first with optional cloud** is the correct default for developer tools, and that the privacy, latency, and cost advantages of local execution outweigh the scalability advantages of cloud-only deployment for the majority of use cases. These bets are falsifiable and the architecture accommodates pivoting if they prove wrong.
