# Usage Tracker Lifecycle

The runtime's `UsageTracker` tracks token consumption across a session in two scopes: per-turn (latest) and cumulative (session-total). It reconstructs state from persisted session history on startup, then updates incrementally as each turn completes.

## TokenUsage: Four Distinct Dimensions

`TokenUsage` in `crates/runtime/src/usage.rs` tracks four independent counters:

```rust
pub struct TokenUsage {
    pub input_tokens: u32,
    pub output_tokens: u32,
    pub cache_creation_input_tokens: u32,
    pub cache_read_input_tokens: u32,
}
```

- **input_tokens**: raw tokens in the turn's request
- **output_tokens**: tokens generated by the model
- **cache_creation_input_tokens**: tokens consumed building a prompt cache
- **cache_read_input_tokens**: tokens served from a cache hit on a subsequent request

These map directly to the Anthropic API's usage reporting surface. The four-field structure is not an implementation artifact—it reflects the provider's own metering.

## Reconstruction on Startup

When `ConversationRuntime::new_with_features` is called in `crates/runtime/src/conversation.rs`, it immediately calls:

```rust
let usage_tracker = UsageTracker::from_session(&session);
```

`UsageTracker::from_session` in `usage.rs` iterates over every persisted `ConversationMessage` in the session and calls `record` for each one that carries usage metadata:

```rust
pub fn from_session(session: &Session) -> Self {
    let mut tracker = Self::new();
    for message in &session.messages {
        if let Some(usage) = message.usage {
            tracker.record(usage);
        }
    }
    tracker
}
```

This means resuming a session is lossless with respect to usage accounting—the tracker has the same cumulative totals after reconstruction that it had before the session was persisted.

## Per-Turn Incremental Recording

During `run_turn`, the runtime's API client emits `AssistantEvent::Usage(TokenUsage)` when the provider delivers usage data. In `build_assistant_message`, this is extracted and passed to:

```rust
if let Some(usage) = usage {
    self.usage_tracker.record(usage);
}
```

`record` updates both scopes:

```rust
pub fn record(&mut self, usage: TokenUsage) {
    self.latest_turn = usage;                                            // current turn
    self.cumulative.input_tokens += usage.input_tokens;
    self.cumulative.output_tokens += usage.output_tokens;
    self.cumulative.cache_creation_input_tokens += usage.cache_creation_input_tokens;
    self.cumulative.cache_read_input_tokens += usage.cache_read_input_tokens; // cumulative
    self.turns += 1;
}
```

`latest_turn` holds only the most recent turn's counters; `cumulative` is a running sum across all turns. The distinction matters for two separate concerns:

- **Current-turn usage** (`current_turn_usage()`) drives mid-session decisions such as auto-compaction threshold checks in `maybe_auto_compact`, which gates on `cumulative_usage().input_tokens`.
- **Cumulative usage** (`cumulative_usage()`) is what surfaces in the turn summary and CLI cost displays as session-total spend.

## Prompt Cache Telemetry

The runtime also handles `PromptCacheEvent` separately from `TokenUsage`:

```rust
pub struct PromptCacheEvent {
    pub unexpected: bool,
    pub reason: String,
    pub previous_cache_read_input_tokens: u32,
    pub current_cache_read_input_tokens: u32,
    pub token_drop: u32,
}
```

`PromptCacheEvent` is captured into the `TurnSummary` alongside usage, allowing callers to observe cache efficiency anomalies (e.g., unexpected cache invalidation causing `token_drop`) without conflating them with the billing counters.

## Builder Lessons

1. **Reconstruction is not optional bookkeeping.** `from_session` exists because usage is a correctness concern for billing and budget enforcement, not just informational metadata. If a session is resumed and usage is not reconstructed, auto-compaction thresholds would be wrong and cost summaries would be incomplete.

2. **Keep latest and cumulative as separate fields.** Storing `latest_turn` separately from `cumulative` makes both scopes trivially accessible and avoids recomputation. A single accumulated counter would require storing a turn history to recover the per-turn value.

3. **Four-dimensional usage tracking is not over-engineering.** `cache_creation_input_tokens` and `cache_read_input_tokens` have different pricing from input/output tokens. A three-field struct that omits cache dimensions would produce incorrect cost estimates for workloads that hit the cache heavily.

4. **Prompt cache events are first-class telemetry, not error logs.** The runtime surfaces cache events through the turn summary rather than panicking or silently absorbing them. This allows callers to observe cache health without runtime infrastructure changes.
